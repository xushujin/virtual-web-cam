# VirtualWebCam 用户使用指南

版本：1.0  
适用对象：项目管理员、现场实施人员、运维人员  
访问地址：`http://<宿主机IP>:9528`，当前示例为 `http://192.168.5.198:9528`

## 1. 系统能做什么

VirtualWebCam 可以把一个网页变成一路视频源。系统支持两种接入方式：

- ONVIF 摄像头：每一路有独立 IP，同时提供 RTSP、ONVIF 和 go2rtc 页面。
- RTSP 流源：多路共享宿主机 IP，通过不同流路径区分，只提供 RTSP，适合现场 IP 地址不足的项目。

创建 ONVIF 摄像头后，你可以得到：

```text
RTSP 地址：rtsp://摄像头IP:554/流名称
ONVIF 地址：http://摄像头IP/onvif/device_service
go2rtc 页面：http://摄像头IP
```

创建 RTSP 流源后，你可以得到：

```text
RTSP 地址：rtsp://宿主机IP:554/流名称
```

中控、大屏系统、ONVIF Device Manager 或播放器可以用这些地址接入。ONVIF Device Manager 只适用于 ONVIF 摄像头。

你不需要手动运行 Xvfb、Chrome、FFmpeg、MediaMTX 或 go2rtc。只需要在网页管理后台中创建摄像头即可。

## 2. 基本概念

### 2.1 项目

项目用于区分不同现场、客户或大屏系统。每个项目可以有不同的矩阵屏幕规格。

示例：

```text
总部大厅项目：6 行 x 8 列，共 48 块屏
门店窄屏项目：2 行 x 8 列，共 16 块屏
```

### 2.2 视频源

一条视频源记录对应一路网页画面。后台列表中仍使用“摄像头管理”作为业务入口，但源类型可以是 ONVIF 摄像头或 RTSP 流源。

视频源包含：

- 名称。
- 源类型。
- 虚拟 IP。仅 ONVIF 摄像头需要。
- 网页 URL。
- 流名称。新增和批量生成时由系统自动生成，通常不需要用户关注。
- 分辨率。
- FPS。
- 运行状态。
- RTSP 地址，以及 ONVIF 摄像头才有的 ONVIF 地址。
- 投放屏幕绑定。

### 2.3 矩阵屏幕

矩阵屏幕用于管理大屏的屏幕编号和绑定关系。它帮助你直观看到：

- 哪一路摄像头投向哪块屏。
- 哪一路摄像头占用了连续几块屏。
- 哪些摄像头还没有绑定屏幕。

### 2.4 围栏区域

如果一个摄像头需要占用连续多块屏，可以在矩阵绑定页面用鼠标画一个矩形围栏。摄像头绑定到围栏后，就表示这一路视频投向这个合并区域。

没有画围栏时，每块屏就是一个独立区域。

## 3. 进入系统

浏览器打开：

```text
http://192.168.5.198:9528
```

页面顶部会显示运行环境状态。

绿色表示：

```text
Docker、macvlan 网络、VirtualWebCam 镜像可用
```

黄色或红色表示环境需要运维处理。常见问题包括：

- Docker 没运行。
- 后端无法访问 Docker socket。
- macvlan 网络不存在。
- `virtualwebcam:latest` 镜像不存在。

### 3.1 登录

系统内置账号登录。首次部署时如果还没有用户，后端会自动创建默认管理员：

```text
用户名：admin
密码：admin123456
```

生产环境应由运维在 `.env` 或启动环境变量中修改 `ADMIN_PASSWORD`。管理员登录后，可以在系统级“用户管理”页面创建登录人员，并授权他们访问指定项目。

普通用户登录后只能看到已授权项目：

- `仅查看`：可以查看项目、摄像头、矩阵、日志和 RTSP / ONVIF 地址。
- `可操作`：可以管理授权项目内的视频源、容器启停、矩阵绑定和项目设置。

## 4. 项目入口

登录后先进入首页。首页用于管理项目；系统管理员能看到所有项目，普通用户只能看到管理员授权给他的项目。

每个项目卡片包含：

- 项目名称。
- 矩阵规格，例如 `6x8`。
- 屏幕编号范围，例如 `屏01 - 屏48`。
- 摄像头管理。
- 矩阵绑定。
- 项目设置。
- 用户管理：系统级入口，仅系统管理员在首页可见，用于维护登录人员和项目资源授权。

### 4.1 创建项目

在“创建管理项目”区域填写：

- 项目名称。
- 行数。
- 列数。
- 屏幕编号前缀。

点击“新建项目”。

建议命名：

```text
总部大厅大屏
一楼指挥中心
门店窄屏项目
```

### 4.2 进入项目

常用入口：

- “摄像头管理”：创建和维护 ONVIF 摄像头、RTSP 流源等视频源。
- “大屏地址”：维护当前项目常用的大屏网页地址。
- “矩阵绑定”：维护摄像头和屏幕区域的关系。
- “项目设置”：修改项目名称、矩阵规格、导入导出。

### 4.3 大屏地址管理

大屏地址是项目级地址库，用来保存当前项目常用的大屏网页 URL。现场创建或编辑视频源时，不需要每次手动复制长地址，可以在网页 URL 输入框中搜索并选择。

进入项目后点击“大屏地址”：

1. 填写名称，例如 `大厅信息屏`。
2. 填写网页地址，例如 `https://example.com/dashboard`。
3. 可选填写备注。
4. 点击“添加地址”。

保存后，在新增 ONVIF 摄像头、新增 RTSP 流源、批量生成和编辑视频源时，网页 URL 输入框会出现这些地址作为候选。仍然可以直接手动输入任意合法的 `http://` 或 `https://` 地址。

只有系统管理员或该项目的“可操作”用户可以新增、编辑和删除大屏地址；“仅查看”用户只能查看和使用地址。

大屏地址页还提供独立的 CSV 导入导出：

- 导出 CSV：导出当前项目的大屏地址，列为 `name,url,remark`。
- 导入 CSV：从本地 CSV 批量新增到当前项目；`name` 和 `url` 必填，`remark` 可选。

CSV 示例：

```csv
name,url,remark
大厅信息屏,https://example.com/dashboard,大厅常用地址
```

## 5. 摄像头管理

摄像头管理页面主要用于查看摄像头详情列表和执行运维操作。

### 5.1 页面信息

顶部统计：

- 总数。
- 运行中。
- 已停止。
- 异常。
- 已绑定。
- 未绑定。

点击统计项可以快速筛选。

列表列信息：

- 名称。
- IP。
- 网页 URL。
- 状态。
- 资源。
- 投放屏幕。
- RTSP。
- ONVIF。
- 操作。

默认是精简列表，只显示最常用的名称、IP、网页 URL、状态和操作。可以在“列表字段”里手动显示或隐藏资源、投放屏幕、RTSP、ONVIF 等长字段，便于按现场排查目标调整表格。

资源监控区域默认折叠为一行摘要，显示 CPU、内存、网络和磁盘写入概览，并继续按后台轮询自动刷新。点击“展开”后显示：

- CPU 合计：当前项目下运行容器的 CPU 使用率合计。
- 内存合计：当前运行容器的内存占用。
- 网络速率：两次采样之间的接收和发送速率，同时保留累计值。
- 磁盘读写速率：两次采样之间的读写速率，同时保留累计写入量。

在“列表字段”中开启“资源”列后，每一行摄像头也会显示该路摄像头的 CPU、内存、网络速率和磁盘读写速率。现场做硬件规划时，建议先启动少量真实网页，观察单路平均消耗，再按目标路数估算整机 CPU、内存和磁盘写入压力。

### 5.2 状态含义

```text
运行中：Docker 容器正在运行
已停止：容器不存在或已停止
异常：容器异常退出、Docker 操作失败或状态不可用
```

运行中的摄像头不能重复点击启动。  
已停止的摄像头不能点击停止。  
重启用于让当前配置重新生效。

### 5.3 新增 ONVIF 摄像头

点击“新增源”，填写：

- 源类型：选择 `ONVIF 摄像头（独立 IP）`。
- 名称：例如 `web-cam-01`。
- 虚拟 IP：例如 `192.168.5.200`。
- 网页 URL：要显示的大屏网页地址。可以手动输入，也可以从项目“大屏地址”候选中搜索选择。
- 宽度：通常 `1280`。
- 高度：通常 `720`。
- FPS：通常 `15`。

从“大屏地址”候选中选择网页 URL 时，系统会自动把该地址名称回填到视频源名称中。流名称由系统自动生成，用于 RTSP 路径和 ONVIF Profile。

点击创建后，系统会：

1. 保存摄像头配置。
2. 创建 Docker 容器。
3. 启动 Xvfb、Chrome、FFmpeg、MediaMTX、go2rtc。
4. 生成 RTSP 和 ONVIF 地址。
5. 在列表中显示状态。

如果 Docker 未运行、Docker socket 无法访问、网络或镜像不可用，创建弹窗会直接显示失败原因。列表中已经写入但启动失败的视频源会显示“异常”，可以打开日志查看原因。

### 5.4 新增 RTSP 流源

如果现场 IP 地址不足，或者中控支持直接 RTSP 接入，可以新增 RTSP 流源。

点击“新增源”，填写：

- 源类型：选择 `RTSP 流源（共享 IP + 流路径）`。
- 名称：例如 `rtsp-screen-01`。
- 网页 URL：要显示的大屏网页地址。可以手动输入，也可以从项目“大屏地址”候选中搜索选择。
- 宽度：通常 `1280`。
- 高度：通常 `720`。
- FPS：通常 `15`。

RTSP 流源不需要填写虚拟 IP。流名称由系统自动生成，并会避免和已有 RTSP 流源冲突。创建后系统会自动维护共享 RTSP 网关，最终地址类似：

```text
rtsp://192.168.5.198:554/screen01
```

RTSP 流源没有 ONVIF 地址，也没有 go2rtc 页面。

### 5.5 批量生成视频源

点击“批量生成”。

批量生成只写入配置，不立即启动容器，适合一次录入多路 ONVIF 摄像头或 RTSP 流源。流名称由系统自动生成，用户不需要填写。

需要填写：

- 源类型。可选择 ONVIF 摄像头或 RTSP 流源。
- 数量。
- 起始 IP。仅 ONVIF 摄像头需要。
- 名称前缀。
- 网页 URL。可以手动输入，也可以从项目“大屏地址”候选中搜索选择。
- 分辨率。
- FPS。

示例：

```text
数量：8
源类型：ONVIF 摄像头（独立 IP）
起始 IP：192.168.5.200
名称前缀：web-cam-
```

生成结果：

```text
web-cam-01 -> 192.168.5.200 -> screen01
web-cam-02 -> 192.168.5.201 -> screen02
...
web-cam-08 -> 192.168.5.207 -> screen08
```

如果源类型选择 RTSP 流源，则不需要填写起始 IP，系统会生成共享网关下的流路径。

批量生成后，可勾选需要启动的视频源，点击批量启动。

### 5.6 搜索和筛选

搜索框支持：

- 摄像头名称。
- IP。
- 流名称。
- 网页 URL。
- 屏幕编号。
- RTSP。
- ONVIF。

状态筛选支持：

- 全部状态。
- 运行中。
- 已停止。
- 异常。

### 5.7 单行快捷操作

每行右侧有快捷按钮：

- 启动。
- 停止。
- 重启。
- 配置。

不可执行的按钮会自动禁用，避免误操作。

### 5.8 配置面板

点击“配置”后，会展开该摄像头的操作面板。

配置面板只会在再次点击当前行“配置”或点击其他行“配置”时收起；启动、停止、重启、查看日志、复制等操作不会自动收起面板，便于连续操作。
如果 Docker socket 不可访问，启动、停止、重启等操作会在页面上显示明确错误提示。

常用操作：

- 启动。
- 停止。
- 重启。
- 查看日志。
- 编辑配置。
- 复制为新摄像头。
- 复制 RTSP。
- 复制 IP。
- 复制 mpv 测试命令。
- 删除摄像头。

RTSP 流源没有 ONVIF 地址，所以 ONVIF 字段不会显示。

### 5.9 编辑视频源

点击“配置” -> “编辑配置”。

可修改：

- 源类型。
- 名称。
- IP。仅 ONVIF 摄像头需要。
- 网页 URL。
- 流名称。一般保持系统生成值；只有需要调整 RTSP 路径时再修改。
- 宽度。
- 高度。
- FPS。

如果摄像头正在运行，保存后系统会尝试重建容器，使新配置生效。

修改网页 URL 后，建议重新打开播放器：

```bash
mpv --rtsp-transport=tcp rtsp://192.168.5.200:554/screen01
```

部分播放器可能停留在旧画面，重新打开即可。

### 5.10 查看日志

点击“配置” -> “查看日志”。

日志可以帮助判断：

- Chrome 是否打开了目标网页。
- FFmpeg 是否正常推流。
- MediaMTX 是否接收到流。ONVIF 模式为容器内 MediaMTX；RTSP 模式为共享网关 MediaMTX。
- go2rtc 是否正常启动。仅 ONVIF 摄像头有 go2rtc。
- 容器是否反复重启。

### 5.11 删除摄像头

点击“配置” -> “删除摄像头”。

删除会同时：

- 删除数据库记录。
- 删除对应 Docker 容器。
- 释放该摄像头绑定的屏幕区域。

删除前请确认该摄像头不再被中控使用。

## 6. 矩阵绑定

矩阵绑定页面用于维护“摄像头”和“大屏屏幕”的关系。

### 6.1 页面结构

左侧：

- 未绑定摄像头源。
- 当前围栏信息。
- 已绑定区域。

右侧：

- 屏幕矩阵。
- 单屏区域。
- 合并围栏区域。

### 6.2 单屏绑定

如果摄像头只投向一块屏：

1. 在左侧找到未绑定摄像头。
2. 拖拽到目标屏幕。
3. 松开鼠标。

绑定成功后：

- 该摄像头从“未绑定摄像头源”列表中移除。
- 屏幕格子中显示摄像头信息。
- 摄像头列表中的“投放屏幕”会显示对应屏幕。

### 6.3 合并区域绑定

如果一个摄像头需要占用连续几块屏：

1. 在矩阵上按住鼠标。
2. 拖出一个矩形区域。
3. 松开鼠标，形成围栏。
4. 把未绑定摄像头拖入这个围栏。

示例：

```text
屏01、屏02、屏03
屏09、屏10、屏11
```

这是一个 `3列 x 2行` 的合并区域。

绑定后会显示：

```text
摄像头名称
屏01 - 屏11 · 3列x2行
IP
流名称
状态
```

围栏保持透明，可以看到被圈住的屏幕编号。
点击已经绑定到矩阵围栏的视频源，可以直接打开编辑窗口，快速修改视频源名称和网页 URL。围栏内的打开按钮会打开该视频源配置的网页 URL，便于现场核对大屏页面。

### 6.4 解除绑定

在已绑定区域或矩阵围栏上点击解除按钮。

解除后：

- 摄像头回到未绑定列表。
- 屏幕或围栏恢复为空闲。
- 可以重新拖拽绑定。

### 6.5 绑定限制

- 一个屏幕同一时间只能绑定一路摄像头。
- 摄像头绑定后不会继续出现在未绑定列表中。
- 多屏合并必须是连续矩形。
- 不支持一个摄像头绑定多个分散区域。
- 不支持两个摄像头共享同一块屏。

这些限制是为了避免现场投屏关系混乱。

## 7. 项目设置

项目设置用于调整项目级信息：

- 项目名称。
- 行数。
- 列数。
- 屏幕编号前缀。
- 导出项目配置。
- 导入项目配置。

### 7.1 修改矩阵规格

修改行列数前需要注意：

- 如果缩小矩阵，超出范围的屏幕绑定可能需要重新整理。
- 建议先解除不确定的绑定，再调整规格。
- 调整后到矩阵绑定页面检查显示是否正确。

### 7.2 导出项目

导出用于备份或迁移项目。

导出内容包含：

- 项目信息。
- 矩阵规格。
- 摄像头配置。
- 大屏地址库。
- 屏幕绑定。
- RTSP 地址，以及 ONVIF 摄像头的 ONVIF 地址。

### 7.3 导入项目

导入时如果发现 ONVIF 摄像头 IP 冲突，系统会尝试在同网段寻找可用 IP，并显示重映射结果。RTSP 流源如果和现有共享流路径冲突，系统会自动改名，例如 `screen01` 可能变成 `screen01-import`。

如果导入文件里的矩阵绑定互相冲突，系统会拒绝导入，并清理已临时创建的项目，避免留下半成品数据。

导入后建议：

1. 检查摄像头 IP。
2. 检查 RTSP 流名称。
3. 检查大屏地址库。
4. 检查矩阵绑定。
5. 分批启动摄像头。
5. 逐路验证 RTSP；ONVIF 摄像头再验证 ONVIF。

## 8. 用户管理

只有系统管理员能在首页看到“用户管理”入口。登录人员是系统级主体，项目只是分配给登录人员的资源。

常用流程：

1. 在首页顶部点击“用户管理”。
2. 创建普通用户，设置用户名、显示名和初始密码。
3. 在左侧选择某个登录人员。
4. 在右侧项目资源列表中给该登录人员选择权限：
   - `不授权`：该用户看不到此项目。
   - `仅查看`：只能查看配置和地址。
   - `可操作`：可以管理该项目内的视频源、矩阵绑定和容器启停。
5. 点击“保存授权”。

授权保存后，普通用户重新登录或刷新页面即可看到自己被授权的项目。

## 10. 接入中控

### 10.1 ONVIF 手动添加

中控支持 ONVIF 手动添加时填写：

```text
协议：ONVIF
IP：192.168.5.200
端口：80
用户名：空
密码：空
```

如果要求完整地址：

```text
http://192.168.5.200/onvif/device_service
```

### 10.2 RTSP 直连

ONVIF 摄像头的 RTSP 地址：

```text
rtsp://192.168.5.200:554/screen01
```

RTSP 流源的 RTSP 地址：

```text
rtsp://192.168.5.198:554/screen01
```

两者的区别是：ONVIF 摄像头使用摄像头独立 IP，RTSP 流源使用宿主机共享网关 IP。

### 10.3 ODM 测试

ONVIF Device Manager 手动添加：

```text
http://192.168.5.200/onvif/device_service
```

成功后应能看到：

```text
Profile：screen01
Stream URI：rtsp://192.168.5.200:554/screen01
Live Video：网页画面
```

## 11. 播放器测试

推荐：

```bash
mpv --rtsp-transport=tcp rtsp://192.168.5.200:554/screen01
```

如果是 RTSP 流源：

```bash
mpv --rtsp-transport=tcp rtsp://192.168.5.198:554/screen01
```

也可以使用：

```bash
ffplay -rtsp_transport tcp rtsp://192.168.5.200:554/screen01
```

如果修改了网页 URL 或重启了容器，播放器可能不会自动刷新。关闭播放器重新打开即可。

## 12. 推荐命名和 IP 规划

### 12.1 IP 规划

当前地址池：

```text
192.168.5.192/26
```

建议：

```text
192.168.5.199  宿主机 macvlan-host 辅助接口
192.168.5.200  摄像头 01
192.168.5.201  摄像头 02
192.168.5.202  摄像头 03
...
```

不要把摄像头 IP 设置成网关、宿主机 IP 或已被其他设备占用的 IP。

### 12.2 摄像头命名

建议名称体现用途：

```text
大厅信息屏
生产看板
会议室日程
西南片区数据屏
```

### 12.3 流名称

新增和批量生成视频源时，流名称由系统自动生成。只有在编辑配置或 API 调用中需要手动调整 RTSP 路径时，才需要关注它。

手动调整时建议使用英文、数字和短横线：

```text
screen01
screen02
lobby-info
meeting-schedule
```

不要使用中文、空格或特殊符号。

## 13. 常见问题

### 13.1 为什么页面显示运行中，但画面是网页错误页？

说明 RTSP 管道正常，但 Chrome 打不开目标网页。请检查网页地址、网络、DNS、证书和登录权限。

### 13.2 修改网页 URL 后为什么播放器还是旧画面？

摄像头容器重启后，部分播放器会停留最后一帧。关闭播放器后重新打开即可。

推荐用 mpv 验收，因为源断开时它通常会退出。

### 13.3 为什么宿主机本机打不开摄像头 IP？

这是 macvlan 的常见限制。运维需要配置 `macvlan-host` 辅助接口。其他同网段机器通常可以直接访问。

RTSP 流源不使用 macvlan 摄像头 IP，不受这个限制；它使用宿主机 IP，例如 `rtsp://192.168.5.198:554/screen01`。

### 13.4 为什么 ONVIF 自动发现不到？

自动发现依赖 UDP 3702 和网络广播策略。当前主要支持手动添加。只要手动添加成功并能看到 Live Video，就说明可接入。

### 13.5 为什么一个屏幕不能绑定多个摄像头？

为了避免现场关系混乱，一个屏幕同一时间只能属于一个摄像头区域。如果需要一个摄像头占用多块屏，请使用矩形围栏合并区域。

### 13.6 批量生成为什么不直接启动？

几十路摄像头同时启动会带来较大 CPU、内存和网络压力。批量生成先写配置，现场可以分批启动，便于观察资源和故障。

### 13.7 换了网络环境或网段后要怎么处理？

如果现场网段从 `192.168.5.xxx` 变成 `192.168.9.xxx`，需要运维重新创建 Docker macvlan 网络、重新配置宿主机 `macvlan-host` 辅助接口，并把管理后台里的摄像头虚拟 IP 改到新网段。

摄像头镜像不用重新构建。RTSP 流源不需要改虚拟 IP，但需要把后端配置里的 `RTSP_GATEWAY_HOST` 改成新的宿主机 IP。详细命令见《部署运维文档》的“更换网络环境 / 网段迁移”章节。

### 13.8 什么时候用 RTSP 流源？

如果项目现场 IP 地址不足，或者中控可以直接按 RTSP URL 接入，建议使用 RTSP 流源。它不占用独立摄像头 IP，只需要保证宿主机 `554/tcp` 可访问。

如果中控必须按 ONVIF 设备添加，或者必须让每一路表现为独立摄像头 IP，就使用 ONVIF 摄像头。

## 14. 日常操作建议

- 创建 ONVIF 摄像头后先用 go2rtc Web 或 mpv 验证。
- 创建 RTSP 流源后直接用 mpv 验证。
- ONVIF 摄像头再用 ODM 手动添加验证 ONVIF。
- 最后接入中控。
- 批量摄像头分批启动。
- 大屏静态页面优先使用 `1280x720@15` 或更低 FPS。
- 修改 URL 后重新打开播放器确认。
- 删除摄像头前确认中控侧已经不再使用。
